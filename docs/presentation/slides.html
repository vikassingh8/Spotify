02 · Architecture: the showcase

End-to-end data flow

React Player→ NGINX Gateway→ Express Services→ Kafka→ PySpark Streaming→ Redis / Postgres / Blob→ Client

Decoupled write path

Playback returns instantly after publishing to Kafka, so analytics happen asynchronously.

Independent scaling

Stateless JSON Web Token (JWT) services scale horizontally behind NGINX / Kubernetes (K8s) load-balancing.

Event buffer

Kafka absorbs spikes; Spark consumes at its own pace, so no backpressure on users.

03 · Real-Time Processing

The streaming pipeline

PySpark Structured Streaming

  • Consumes play-events topic every 5s (micro-batch)
  • Anonymizes user id with Hash-based Message Authentication Code (HMAC) before any analytics
  • Windows events per minute, aggregates plays per song / genre
  • Writes leaderboards → Redis, rollups → Postgres

Why it's robust

  • Parallelism: Kafka partitions × Spark executors
  • Fault tolerance: checkpointing resumes from last offset
  • Freshness: trending decays via per-minute Time-To-Live (TTL) Redis buckets

05 · Security & Privacy

Defense in depth

Authentication

bcrypt password hashing + JWT sessions · auth rate-limiting · admin-gated ingestion controls.

Authorization: Role-Based Access Control (RBAC)

free / premium / admin roles · free user gets 403 on a premium track (verified).

Encryption

Transport Layer Security (TLS) in transit at gateway · encrypted Blob / Postgres / Redis at rest.

Privacy by design

HMAC pseudonymization inside the pipeline: raw user id dropped before analytics (verified: 0 raw IDs downstream). Data minimization · General Data Protection Regulation (GDPR) / California Consumer Privacy Act (CCPA) alignment (erasure, minimization).

07 · Trade-offs

Engineering decisions

CAP theorem (Consistency, Availability, Partition tolerance)

Analytics are eventually consistent (a few seconds behind) to keep the write path fast & available.

Latency vs durability

Async processing adds slight delay but guarantees no lost events (Kafka buffer).

Cost vs performance

Reserved / spot capacity vs always-on headroom.

Challenges solved

At-least-once vs exactly-once semantics · coordinating an async pipeline with a responsive UI · Spark–Kafka jar provisioning (warmed Ivy cache at build).

Future work

CDN for audio · Machine Learning (ML) recommender (collaborative filtering) · multi-region active-active · exactly-once idempotent sinks · OpenID Connect (OIDC) via Entra ID.
